## Findings（按嚴重度）

### HIGH-1 — Signature malleability（已實測復現）✅ 已修 2026-07-09
> 修復：`parse_signature_hex` 拒絕 s > N//2（EIP-2 low-s）；regression 在 selftest part C（高-s 變體 parse 拒收 + verify 返 False）。基準 182 → 184。
`eiv/eth.py:245-256`（parse_signature_hex）+ `eiv/eip712.py:175-184`（verify）
verify 不檢查 low-s：`(r, N-s, v^1)` 對原 signer 仍回 True → 同一授權有兩個有效簽章。若上層或未來 on-chain 層用 signature bytes / intent_hash+signature 當去重或 replay key，可被繞過。ecdsa_sign 有 low-s normalization（eth.py:214）但驗證端沒強制。修法：parse_signature_hex 或 verify 拒絕 s > N//2。

### HIGH-2 — max_slippage_bps 進簽章卻從不 enforce
`eiv/predicates.py:108-116`
欄位被 canonicalize、hash、簽進 EIP-712 digest（eip712.py:122），但 validate() 只比 amount_in/out 邊界。user 簽 0.5% 滑點上限、實際 5% 滑點仍可 PASS（只要 amount_out >= min_amount_out）。predicates.py 是凍結契約 → 決策：enforce（動凍結契約，要 user 拍板）或文件明講不 enforce。

### MEDIUM-1 — RPC trace 多 topic 事件誤算 amount（PLAUSIBLE）
`eiv/chain_adapter.py:246-259`
只用 len(topics)<3 過濾；ERC-721 Transfer 等 data 語意不同的事件可能被當 ERC-20 金額累加，污染 amount_in/out。未實測觸發。

### MEDIUM-2 — recover_intent_signer 的 chain_id=None 預設
`eiv/eip712.py:148-154`
省略 chain_id 落到 env 預設（Sepolia），會在錯 domain 下 recover。主路徑安全（verify 有明確傳 chain_id，跨鏈簽章實測被拒），是 public helper 誤用風險。建議強制帶參數。

## 明確沒問題的區域
Domain separation（跨鏈 replay 實測被拒）、Keccak/secp256k1/RFC6979（有 published vectors + round-trip）、路徑穿越（basename+黑名單）、SQL injection（全 parametrized）、secrets 不進 log、無 debug endpoint、CORS * 為刻意可接受。
